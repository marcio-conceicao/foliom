    - timeout-minutes: 5

    Steps:
    1. actions/checkout@v4
    2. Install Rust stable toolchain + rust cache (same pattern as existing jobs; share cache key phase4-watcher-smoke).
    3. Install cargo-nextest via taiki-e/install-action@nextest (same as existing test job).

    6. Start the server in background: ./target/release/foliom serve /tmp/watcher-smoke &; FOLIOM_PID=$!
    7. Subscribe to SSE: curl --silent --no-buffer -N http://127.0.0.1:7345/api/watch/events > /tmp/sse_output.txt & ; SSE_PID=$!
    8. Wait for server warmup: sleep 1
    9. Write external file: echo "- ExternalEdit $(date)" > /tmp/watcher-smoke/pages/ExternalEdit.md
    10. Wait for debounce + coalesce + margin: sleep 1.5
    11. Assert SSE received: grep -q "pages_updated" /tmp/sse_output.txt || (echo "FAIL: no pages_updated event in SSE output"; cat /tmp/sse_output.txt; kill $FOLIOM_PID $SSE_PID 2>/dev/null; exit 1)
    12. Print success: echo "OK: pages_updated received"; cat /tmp/sse_output.txt
    13. Cleanup: kill $FOLIOM_PID $SSE_PID 2>/dev/null; true

    Also add a step to run the watcher_integration unit tests that were created in 04-01 (this ensures the integration tests also run in the existing test job — verify they are already added to the workspace test suite by checking if the test job's `cargo nextest run --workspace` step covers them; if not, add an explicit `cargo nextest run -p foliom-cli --test watcher_integration` step to the existing test job).

    WSL2 developer caveat note: The CI job runs on ubuntu-latest (real Linux kernel), not WSL2. The job will pass even if WSL2 inotify for /mnt/c/ paths has limitations — CI exercises native Linux inotify.

    ## Purpose
    Manual acceptance checklist for Phase 4 — Disk Sync scenarios that cannot be fully automated in CI: Windows watcher behavior and Syncthing-storm simulation.

    ## Acceptance Scenarios

    ### Scenario 1: VS Code atomic-rename save (macOS / Linux / Windows)
    Prerequisites: foliom serve running against a real notes folder.
    Steps:
    1. Open a .md file in VS Code.
    2. Edit and save (Cmd/Ctrl+S).
    3. Expected: The page reloads in the Foliom browser tab within ~1s. No duplicate SSE events. No "File changed externally" banner if no block was in edit mode.
    Accept criteria: Page content in browser matches saved file content.

    ### Scenario 2: Syncthing-style storm (Linux / macOS)
    Prerequisites: foliom serve running; a notes folder with 10+ .md files.
    Steps:
    1. Open browser dev tools → Network → filter by /api/watch/events.
    2. Run: `for i in $(seq 1 10); do echo "- Sync $i $(date)" > notes/pages/sync_$i.md; done`
    3. Expected: Within 1s, exactly 1 (or at most 2) `pages_updated` SSE events arrive — not 10. The browser tab shows the updated page content.
    Accept criteria: Event count in Network tab is 1–2; no UI freeze; no runaway reindex log entries.

    ### Scenario 3: Windows ReadDirectoryChangesW recovery
    Prerequisites: Windows 11 native (not WSL2); foliom serve running.
    Steps:
    1. Start foliom serve and open the browser.
    2. Run a `git checkout` that modifies 100+ .md files simultaneously (simulate large pull).
    3. Expected: foliom server log shows "watcher error — treating as potential missed events" followed by "full reindex triggered". Browser tab shows index_reset event in Network tab, then reloads the current page.
    Accept criteria: Server continues running (no crash); page content updated after reload; watcher re-armed (subsequent single-file edits trigger pages_updated normally).

    ### Scenario 4: Conflict banner (SNC-06)
    Prerequisites: foliom serve running; browser tab open on a page.
    Steps:
    1. Click a block to enter edit mode (CM6 editor appears).
    2. Without saving, edit the same .md file externally (VS Code, echo, etc.).
    3. Expected: A non-blocking banner appears above the editor: "File changed externally. [Reload discards your edit]".
    4. Continue typing in the editor — edit should be possible.
    5. Click Reload — editor closes, page reloads with external content.
    Accept criteria: Banner appears within ~1s of external edit. Continuing to type is unblocked. Reload restores external content. Saving after banner also shows the 409 conflict path from Phase 3.

    ### Scenario 5: Self-write suppression
    Prerequisites: foliom serve running; browser tab open on a page.
    Steps:
    1. Click a block, edit text, blur (or Enter) to save.
    2. Expected: NO "File changed externally" banner appears after saving. The watcher log should show "self-write suppressed" or equivalent.
    Accept criteria: Banner does not appear. Page does not reload after own-write.

    ## Sign-Off Table

    | 3. Windows RDC recovery | Windows 11 | [ ] | | |
    | 4. Conflict banner | Linux/WSL2 | [ ] | | |

    ## Notes
    - WSL2 caveat: inotify works for files on the WSL2 filesystem (/home/...). For notes on the Windows filesystem (/mnt/c/...), notify may fall back to polling or miss events — see PERF-BASELINE.md WSL2 notes.
    - Windows watcher re-arm: after a ReadDirectoryChangesW error, the watcher calls debouncer.watcher().watch() again. Check server log for "re-armed watcher" message.
