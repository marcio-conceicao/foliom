Implement `crates/core/src/storage/location.rs` exactly as in RESEARCH §DB Location + Path Normalization → DB location resolver. The function signature is `pub fn resolve_db_path(notes_root: &Path) -> Result<PathBuf, StorageError>`.

Key implementation points:
1. Canonicalize `notes_root` first (`abs = notes_root.canonicalize()?`).
2. Convert to a UTF-8 string with `to_string_lossy()` and replace `'\\'` → `'/'` for cross-platform hash stability.
3. NFC-normalize the string before hashing: `nfc().collect::<String>()` (so macOS NFD vs Linux NFC produce the same DB filename).
4. `let hash = blake3::hash(nfc.as_bytes());` then take `hash.to_hex()[..16]`.
5. Platform-specific `data_dir()` function:
   - Linux: check `XDG_DATA_HOME`; if unset OR empty (POSIX), fall back to `$HOME/.local/share`.
   - macOS: `$HOME/Library/Application Support`.
   - Windows: `%LOCALAPPDATA%`.

6. `std::fs::create_dir_all(&foliom_dir)?` before returning.

|----------|-------------|
| `notes_root` argument → resolve_db_path | User-provided path; could be a symlink, very long, or non-UTF-8. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | DB-in-notes-folder corruption from cloud sync | mitigate | Resolver places DB in platform data dir, never under `notes_root`. Test asserts `!resolved.starts_with(notes_root)`. (PITFALLS §DB-em-folder-sync.) |
| T-04-02 | Tampering | SQL injection via path string | mitigate | All inserts/queries use parameterized statements (`?` bindings). No string concatenation into SQL anywhere. |
| T-04-03 | Tampering | Migration replay corrupting existing DB | mitigate | `rusqlite_migration` uses `PRAGMA user_version` as the migration cursor; re-running `to_latest` on an already-migrated DB is a no-op. Integration test reopens and asserts idempotence. |
| T-04-04 | Information disclosure | DB world-readable under `$XDG_DATA_HOME` | accept | Single-user local app; permissions follow the user's umask. No secrets stored in the DB (Phase 1). Document for Phase 5 if desktop packaging revisits. |
| T-04-05 | Denial of service | `PRAGMA journal_size_limit` not honored → WAL grows unbounded | mitigate | Explicit `PRAGMA journal_size_limit = 67108864` (64 MB cap) per RESEARCH §PRAGMA setup. `wal_autocheckpoint = 1000` keeps it pruned. |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | New deps: `rusqlite = "0.39"` with `bundled-full`, `rusqlite_migration = "1.3"`, `blake3 = "1.5"`, `tracing = "0.1"`, `tempfile = "3"` (dev). All HIGH legitimacy — locked by CONTEXT D-11/D-16/D-18/D-20. No `[ASSUMED]`/`[SUS]` packages. The `bundled-full` feature pulls in SQLite C source compiled by `cc` — Windows users need MSVC Build Tools (documented in RESEARCH §Environment Availability). |
</threat_model>
