Key steps:

1. Load known files: `let mut known: HashMap<RelativePath, (i64, i64, u64, [u8;32])> = HashMap::new();` populated from `SELECT id, path, mtime_ns, size, hash FROM files`.

2. Walk: `for entry in scanner::walk(root, &IgnoreSet::default_logseq())`.

3. For each entry: compute `rel = RelativePath::from_filesystem(&entry.path, root)?`. Insert into `seen` set. Decide path per the match arms in RESEARCH:
   - Incremental + match on `(mtime_ns, size)` → `stats.unchanged += 1`, continue without read.
   - Otherwise: `fs::read(&entry.path)?`, `blake3::hash(&bytes)`.
   - If hash matches known but mtime differs → `update_file_mtime` only; `stats.mtime_touched += 1`.
   - If file existed and content changed → `reparse_file_tx`; `stats.modified += 1`.
   - If file is new → `insert_file_tx`; `stats.added += 1`.

4. After the walk: iterate `known.keys()` not in `seen`; call `delete_file_cascade`; `stats.deleted += 1`.

5. **One transaction per file.** Open `tx = db.transaction()?`, run the inserts, `tx.commit()?`. This is AP-5 — do NOT wrap the whole reindex in one tx.

6. `:hidden` extension: at startup, try `read_hidden(root.join("logseq/config.edn"))`; if non-empty, `ignore.extend_from_config_edn(hidden)` before walking.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Disk → indexer | Notes folder content is user-controlled; could include huge files, deeply nested bullets, malformed UTF-8, pathological code fences. |
| Indexer → SQLite | All writes go through prepared statements via rusqlite; no dynamic SQL composition. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | SQL injection via filename or block raw | mitigate | All inserts use `?` parameters; rusqlite escapes by binding, never by concatenation. Code review enforces no `format!("INSERT ... '{}' ...")` patterns. |
| T-06-02 | Denial of service | One giant file (100MB markdown) | accept | Memory cost is one file at a time per AP-5; SQLite handles large `raw` columns transparently up to ~1GB default. Phase 2 will add a soft limit if needed. |
| T-06-03 | Tampering | TOCTOU between scanner stat and read | accept | Single-user local app; tolerable. Hash mismatch on next reindex catches divergence. (Documented in RESEARCH §Security Domain.) |
| T-06-04 | Denial of service | Mid-reindex `kill -9` | mitigate | Per-file transactions per AP-5 — completed files are durable, in-flight file rolls back, next reindex re-detects via mtime/hash mismatch. |
| T-06-05 | Information disclosure | Errors leak absolute path into logs | accept | `tracing::warn!` includes paths — appropriate for a local single-user app where the user owns the logs. Phase 5 desktop will revisit log redaction. |
| T-06-06 | Tampering | Deleting DB during reindex | accept | rusqlite holds an open file handle; on Unix the file is unlinked but accessible until close, on Windows the delete will fail. Either way, in-flight operation completes or errors; user data on disk is untouched. |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | New deps: `filetime = "0.2"` (dev-only, standard ecosystem dep, HIGH). No `[ASSUMED]`/`[SUS]` packages. All hot-path crates (`rusqlite`, `blake3`, `walkdir`, `pulldown-cmark`, `unicode-normalization`) already vetted in earlier plans. |
</threat_model>
