Existing signature (do not change):
```rust
pub fn segment(source: &[u8]) -> Vec<RawBlock>;
```

Invariants the implementation MUST satisfy (load-bearing — RESEARCH §Stage 1):
1. Byte ranges contiguous and non-overlapping when sorted by `byte_offset`.
2. `concat(source[b.byte_offset..b.byte_offset+b.byte_length] for b in blocks) == source` exactly.
3. Page prelude block exists (depth = u8::MAX); `byte_length = 0` if the file starts with a bullet at byte 0.
4. Code fences inside bullets stay inside the parent block; the fence-close line is part of the parent's `byte_length`.
5. Drawer contents (`:LOGBOOK:` ... `:END:`) stay inside the parent block; `RawDrawer.byte_offset/byte_length` describe the drawer slice, which is fully contained in the parent's slice.
6. Properties (`key:: value`) are appended to `RawBlock.properties` AND remain verbatim inside `raw`.

  <behavior>
    - Empty input → returns one prelude block with `byte_length = 0`, OR returns an empty Vec (planner picks one — document the choice; the round-trip test accepts both because concat is `""` either way).
    - Input "hello\n" (no bullet) → one prelude block covering all 6 bytes.
    - Input "- foo\n" → prelude (len 0) + one bullet block depth 0, byte_length 6.
    - Input "- foo\n\t- bar\n" → prelude (0) + bullet depth 0 (len 6) + bullet depth 1 (len 7).
    - Input "- foo\n  cont\n- bar\n" → prelude (0) + bullet depth 0 (len 12, includes the 2-space continuation line) + bullet depth 0 (len 6).
    - Input with `:LOGBOOK:\n...\n:END:` inside a bullet → drawer captured as RawDrawer; drawer bytes fully inside parent block's range; parent `byte_length` covers `:END:\n`.
    - Input with `key:: value` continuation line inside a bullet → property pushed to `RawBlock.properties`; line stays in `raw`.
    - Input with code fence opened on the bullet line itself (e.g. `- ```rust\n...\n```\n`) → fence state opens; subsequent lines bypass bullet detection until matching close fence; close-fence line is part of parent block.
    - Input with code fence opened on a continuation line (`  ```\n...\n  ```\n`) → same handling, fence_indent computed from continuation indent.
    - Blank lines between continuation lines belong to the current block (preserve blanks-inside-multi-line-fence and blanks-between-paragraphs).
    - Block depth = TAB count before `- `. A bullet line with leading spaces (not TABs) is NOT a bullet — it's a continuation of the previous block (assumption A3 from RESEARCH).
  </behavior>
  <action>Implement the state machine from RESEARCH §Stage 1 — Line-based Segmenter (concrete design). The pseudocode there is the spec; translate it line-by-line into Rust.

Implementation guidance (no fenced code blocks here — all code lives in segment.rs):

1. Scan `source` line-by-line, tracking `line_start: usize` byte offset of each line. Use `memchr::memchr` for newline finding (add `memchr = "2"` to `[dependencies]` of `crates/core/Cargo.toml` — it is a standard transitive dep of many cargo packages and is pure-rust). Alternatively iterate via `source.split(|&b| b == b'\n')` and recompute offsets manually.

2. State variables: `state: ParserState` enum with variants `InBullet { fence_indent: Option&lt;usize&gt;, drawer: Option&lt;(String, usize)&gt; }` and `InPrelude`. `current_block: Option&lt;RawBlock&gt;`. Output `Vec&lt;RawBlock&gt;` always begins with the prelude block.

3. Bullet detection regex: `^(\t*)- (.*)$`. Match TAB count = depth. Implement by counting leading `b'\t'` bytes manually (no `regex` dep needed at this layer — keep deps minimal in `core`).

4. Continuation prefix: `"\t" * depth + "  "` (two ASCII spaces). Check via byte-slice prefix comparison.

5. Code-fence detection: a line is a fence-opener if, after stripping the bullet-or-continuation prefix, the remaining bytes start with three or more backticks (`\`\`\``). Capture `fence_indent = byte position of first backtick relative to line_start`. Fence-close: a line at the same `fence_indent` containing only optional whitespace + matching backtick count + optional whitespace. For Phase 1 simplicity, accept any line whose trimmed content is exactly three or more backticks of the matching count.

6. Drawer detection: a continuation line whose trimmed content matches `^:([A-Z]+):$` opens a drawer. The drawer's `byte_offset` is `line_start`. `:END:` (case-sensitive, on its own line) closes it. Drawer `byte_length` = bytes from opener line through `:END:\n` inclusive.

7. Property detection: a continuation line whose stripped content matches `^([A-Za-z][A-Za-z0-9._-]*):: (.*)$` is a block property. Push `(key, value)` onto `current_block.properties`. The line ALSO stays in `raw` — properties are tracked in BOTH places per D-05 + the invariants.

8. Emit logic: on bullet-line detection, emit `current_block` (if Some) to the output Vec, then start a new block. On EOF, emit the final block. The prelude block is pushed first; if no bullets exist, it covers the entire file.

9. `raw` field: build by `String::from_utf8(source[b.byte_offset..b.byte_offset + b.byte_length].to_vec()).expect("UTF-8 corpus per A2")`. If you want to handle non-UTF-8 defensively, use `String::from_utf8_lossy().into_owned()` — but document the choice. Assumption A2 says the corpus is UTF-8; the round-trip test will surface any leak.

10. NEVER write a `serialize` / `to_markdown` / `format_block` companion function. Per AP-2 in RESEARCH §Anti-Patterns, the existence of such a function is a future foot-gun. The only way to reconstruct text is `&source[b.byte_offset..b.byte_offset + b.byte_length]`.

After implementing, manually trace the algorithm against `data-folder-sample/Logseq/journals/2023_11_09.md` per RESEARCH §Stage 1 → "Validated against ...". fn assert_splice_noop(source: &[u8], blocks: &[RawBlock]) {

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Denial of service | Pathologically deep TAB nesting (e.g., 10000 TABs before a `- `) | accept | `depth: u8` caps at 255; beyond that, behavior is to treat as continuation (no panic). Document the cap; corpus does not exceed ~5 levels. Phase 2 may add an explicit limit. |
| T-02-02 | Denial of service | Huge unclosed code fence (file is one open fence) | accept | Segmenter scans linearly O(n); no quadratic behavior. Whole file becomes one bullet block. Splice-noop still holds. |
| T-02-03 | Tampering | Non-UTF-8 bytes in source | mitigate | `String::from_utf8` returns Result; segmenter uses `.expect()` with a clear message ("non-UTF-8 source — assumption A2 violated") so the failure mode is loud, not silent corruption. Phase 5 (Tauri) revisits with `from_utf8_lossy` if real-world bases surface. |
| T-02-04 | Tampering | Drawer with no `:END:` (malformed file) | mitigate | Open-drawer state persists to EOF; segmenter emits the drawer with `byte_length` reaching EOF. Splice-noop still holds; user's malformed file is preserved as-is. |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | Only new dep is `memchr = "2"` (or skipped entirely if planner chooses manual newline scanning). `memchr` is a Rust ecosystem cornerstone — direct dep of `regex`, `tokio`, `serde_json`. HIGH legitimacy. No `[ASSUMED]`/`[SUS]` packages. |
