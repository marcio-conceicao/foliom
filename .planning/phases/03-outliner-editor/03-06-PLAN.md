<objective>
Implement the rename + create-page surface with the write-ahead journal that makes backlinks rewrite atomic across SQL + N file rewrites. SNC-05 ("rewrites all `[[oldname]]` references in one atomic transaction; backlinks survive") demands the journal pattern from 03-RESEARCH §3 because the FS is not transactional. LNK-04 ("unresolved link click offers to create") becomes silent create per D-30-03.

Purpose: The single highest-correctness-risk operation in Phase 3 — a partial rewrite would leave the user's notes in a state where searching for `[[oldname]]` returns hits the index claims do not exist. The journal makes the operation idempotent and crash-recoverable.
Output: Backend `crates/core::rename` module with WAL journal; two new HTTP endpoints; frontend `PageHeader` click-to-edit + `RenameModal` + unresolved-link click handler. Tests pin happy path, collision, case-only Windows quirk, and `kill -9` recovery.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP rename request → SQL txn → FS writes → disk rename | Multi-stage write that must be atomic from the user's perspective |
| journal file on disk → boot recovery | Trust boundary on read: malformed journal must not crash the process |
| user-controlled `newName` → filesystem path | Must be validated against reserved chars |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-20 | Tampering | Partial rewrite leaves notes inconsistent across SQL + N files | mitigate | WAL journal pattern per 03-RESEARCH §3. Replay-on-boot is idempotent (each op checks `current_bytes == old_bytes` before applying; `== new_bytes` means already applied → mark + skip). |
| T-03-21 | Tampering | User-controlled `newName` contains path-traversal characters (`../`, NUL) | mitigate | Validate against `RelativePath` constructor (NFC + reject path separators) + explicit Windows reserved-chars list. 400 on rejection. |
| T-03-22 | Information Disclosure | Journal contains raw file paths and byte contents of `[[oldname]]` occurrences | accept | Journal lives in `$XDG_DATA_HOME/foliom/` (user's own data dir; mode 0600 on unix). Contents are byte ranges from files the user already owns. Single-user local app: no cross-user threat. |
| T-03-23 | DoS | 10,000-file rename hangs the request for 30s+ | accept | 03-RESEARCH Open Q2 — blocking by default; if real >100-file renames appear in dogfooding, Phase 4 adds SSE streaming. Single-user app, low risk. |
| T-03-24 | Tampering | Case-only Windows rename silently no-ops because FS treats as same file | mitigate | Two-step rename via `__foliom_rename_tmp__` per 03-RESEARCH Pitfall (case-only). Integration test pins on `#[cfg(windows)]`. |
| T-03-25 | Repudiation | Journal replay applies an op the user manually fixed in another editor | mitigate | Each op checks `current_bytes == old_bytes` before splicing. If neither `old` nor `new` matches, log warning, mark op `applied: skipped`, surface in `RenameResult.warnings`. |
| T-03-SC | Tampering | `regex` crate already in workspace from Phase 1 — no new supply chain | mitigate | No new external crates in this plan. `serde_json` already a Phase 1 dep. |
</threat_model>
