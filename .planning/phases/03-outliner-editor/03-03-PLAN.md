    `put_block` handler signature:
    ```rust
    async fn put_block(
        State(app): State<AppState>,
        Path(block_id): Path<i64>,
        Json(req): Json<PutBlockRequest>,
    ) -> Result<Json<MutationResponse>, ApiError>;

    struct PutBlockRequest { raw: String, prev_hash: String /* hex */ }
    ```

    Run in `tokio::task::spawn_blocking` to satisfy the "DB work off the runtime" pattern from Phase 2.

    Body of the handler (per 03-RESEARCH §7):
    1. Lock `db` (mutex from Phase 2 AppState).
    2. Open SQL transaction.
    3. SELECT the block: `block_id, file_id, byte_offset, byte_length, page_id`. 404 on missing.
    4. SELECT `files.hash, files.path` for that file_id.
    5. Compare `prev_hash` (decoded from hex) to `files.hash`. If mismatch → ROLLBACK + return 409.
    6. Read `original_bytes = fs::read(&abs_path)`.
    7. Compute `new_bytes = splice_block(&original_bytes, byte_offset as usize, byte_length as usize, req.raw.as_bytes())`.
    8. `let new_file_hash = atomic_write_md(&abs_path, &new_bytes, &app.self_writes)?;` (this registers the hash BEFORE the rename per plan 03-01).
    9. UPDATE `blocks SET raw = ?, byte_length = ?` for this block.
    10. For every downstream block in the same file (`byte_offset > original_byte_offset`): UPDATE `byte_offset = byte_offset + (new_len - old_len)` (use `compute_shifted_offsets` math — or a single SQL `UPDATE blocks SET byte_offset = byte_offset + ? WHERE file_id = ? AND byte_offset > ?`).
    11. UPDATE `files SET hash = ?` with `new_file_hash`.
    12. DELETE from `refs WHERE source_block = ?` and re-INSERT via `extract_refs(&req.raw)` mapped through the existing ref-insert helper from indexer.
    13. COMMIT.
    14. Reuse `assemble_tree(&conn, page_id)` to build the response subtree.
    15. Return `MutationResponse { block_subtree, file_hash: hex(new_file_hash), dirty_block_ids: vec![block_id] }`.

    `MutationResponse` and `ApiError` go in `dto.rs`. `ApiError` is a thin enum that maps to status codes (400 BadRequest, 404 NotFound, 409 Stale { current_file_hash: String }, 500 Internal).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP (`127.0.0.1` only — Host allowlist from Phase 2 plan 02-01) → mutation handlers | Untrusted JSON body crosses here |
| handler → filesystem via `atomic_write_md` | Foliom's first write surface in v1 |
| handler → SQL transaction | Same connection as Phase 2 read handlers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Tampering | Stale client clobbers a file edited externally | mitigate | `prev_hash` field required in every mutating request; 409 + `current_file_hash` on mismatch; ROLLBACK before any write. |
| T-03-07 | Information Disclosure | Path traversal via crafted block_id / page_id JSON | mitigate | Block lookup is by integer SQL id; the absolute file path comes from `files.path` (already NFC + forward-slash from Phase 1) joined under `state.root`. No user-controlled path component crosses the boundary. |
| T-03-08 | Tampering | Foliom injects metadata into .md (violates D-13 / PRD §5.6) | mitigate | "No-id-injection" integration test asserts no `id::` or `((uuid))` sequences appear in the file post-mutation that weren't present pre-mutation. Backend NEVER emits these — only the user can type them. |
| T-03-09 | DoS | Unbounded `raw` body crashes the server | mitigate | `axum`'s default request body limit (2 MB) applies. Blocks are typically <10 KB; 2 MB is generous. No per-handler tightening needed in v1. |
| T-03-04 | Tampering | Splice math error corrupts neighbor blocks | mitigate | Inherited from plan 03-02's offset tests + integration "byte-identical neighbors" assertion in Task 1. |
</threat_model>
