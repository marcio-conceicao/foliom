<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| autocomplete query string → SQL LIKE | User-controlled prefix |
| clipboard text → `detectBulletTree` → `postBlock` body | User-controlled text crossing into write path |
| treeOpLog inverse → mutation endpoint | Client-side restoration of state; backend re-validates via `prev_hash` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-15 | Tampering | SQL injection via `prefix` param | mitigate | `params![]` binding; LIKE wildcards `%`/`_` escaped via `\\` with `ESCAPE '\\'` clause. Same pattern as Phase 2 search.rs. |
| T-03-16 | DoS | Pathological autocomplete prefix (empty + high limit) | mitigate | Limit clamped to 100 server-side; default 20. No regex; pure prefix LIKE. |
| T-03-17 | Tampering | Pasted clipboard contains malicious content that bypasses CM6 escaping | accept | Clipboard text is treated as RAW markdown — the same trust level as user typing. Phase 2's renderer already escapes via `markdown-it({ html: false })`. No new XSS surface. |
| T-03-18 | DoS | Huge clipboard paste (10k bullets) blocks UI on `postBlock` loop | mitigate | postBlock loop is sequential `await`s; first ~50 land within 500ms. If user pastes truly massive content, browser becomes slow but does not crash. Document as known v1 limitation; if real, follow-up plan adds a batch endpoint. |
| T-03-19 | Tampering | treeOpLog inverse races with concurrent edits | mitigate | `applyInverse` includes `prev_hash`; on 409 the op is restored to the log and the user sees the Reload banner — same conflict-detection model as plan 03-03. |
</threat_model>
