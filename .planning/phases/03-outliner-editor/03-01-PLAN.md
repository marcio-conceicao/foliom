<objective>
Establish the on-disk write primitive that EVERY edit in Phase 3 will route through, plus the self-write hash set the Phase 4 watcher will consume to suppress its own echo. This is the lowest layer of Phase 3 — nothing else in this phase writes to disk without going through these two modules.

Purpose: SNC-02 demands atomic temp+rename writes with a self-write hash registry. Without this, the round-trip ACPT-01 gate cannot survive real edits (a half-written file would be visible to readers), and Phase 4's watcher would loop on Foliom's own writes.
Output: `crates/core::sync` module exposing `atomic_write_md` + `SelfWriteSet`; tests covering happy path, cross-FS rejection, no-op byte-identity, and (Windows-only, `#[cfg(windows)]`) AV-style retry behavior. Workspace gains `dashmap = "6"` and `tempfile` is promoted from dev-dep to runtime dep in `crates/core`.
</objective>

    Implement `atomic_write_md` per 03-RESEARCH §2 code example. Concretely:

    1. Compute `let hash: [u8;32] = blake3::hash(contents).into();`
    2. `self_writes.register(hash);` (BEFORE the rename to close the race)
    3. `let parent = target.parent().ok_or_else(|| io::Error::new(InvalidInput, "no parent"))?;`
    4. `let mut tmp = tempfile::NamedTempFile::new_in(parent)?;`
    5. `tmp.write_all(contents)?; tmp.as_file().sync_all()?;`
    6. Call `persist_with_retry(tmp, target)` which loops up to 3 attempts with exponential backoff (50ms, 100ms, 200ms) on Windows `PermissionDenied`/`Other`. On unix, no retry — fail fast.
    7. `#[cfg(unix)]`: open the parent dir and `sync_all()` for crash-safety on ext4 without barriers (best-effort, log on failure).
    8. Return `Ok(hash)`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| in-process Rust → OS filesystem | `tempfile::NamedTempFile::persist` is the only path that creates/replaces user `.md` files |
| in-process Rust → in-memory hash set | SelfWriteSet entries cross the boundary between mutation handler (Phase 3) and watcher (future Phase 4) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Partial write visible to readers | mitigate | `tempfile::persist` is atomic on Linux (`rename(2)`) and Windows (`MoveFileExW`); temp file is in the same directory as target → intra-FS rename → atomic. Cross-FS case fails fast with `CrossesDevices` (caller surfaces 500). |
| T-03-02 | Denial of Service | Windows antivirus locks target during `MoveFileExW` | mitigate | 3-retry exponential backoff (50/100/200ms) on `PermissionDenied`. After exhausting retries, return `io::Error` to caller → user sees a "save failed, retry?" UI in plan 03-03. |
| T-03-03 | Repudiation | Foliom's own write is mistaken for an external edit by a future watcher | mitigate | `SelfWriteSet::register(blake3(contents))` BEFORE `persist`. TTL = 30s default (long enough for slowest Defender hold). Phase 4 watcher will call `take_if_present` to suppress its own echo. |
| T-03-SC | Tampering | `dashmap` and `tempfile` supply-chain | mitigate | Both are first-tier crates.io packages with multi-year track records (3+ for dashmap, 8+ for tempfile). 03-RESEARCH §Package Legitimacy lists both as Approved. Lockfile pin captures exact versions. |
</threat_model>
