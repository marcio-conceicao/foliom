**Step A — Create scripts/footprint_check.sh:**

This script is called from CI after `tauri-action` completes. It is bash-only (macOS + Linux).
Windows uses inline PowerShell (see Step B). Make the file executable (`chmod +x`).

```bash
#!/usr/bin/env bash
# scripts/footprint_check.sh
# Installer size gate for DSK-03.
# Usage: bash scripts/footprint_check.sh <installer_path> [ceiling_mb]
# Args:
#   installer_path  — path to .dmg or .msi/.exe installer artifact
#   ceiling_mb      — size ceiling in MB (default: 30)
# Exit code: 0 = within budget, 1 = exceeded
set -euo pipefail

INSTALLER="${1:?Usage: footprint_check.sh <installer_path> [ceiling_mb]}"
CEILING="${2:-30}"

if [ ! -f "$INSTALLER" ]; then
    echo "::error::Installer not found: $INSTALLER"
    exit 1
fi

SIZE_MB=$(du -sm "$INSTALLER" | cut -f1)
echo "Installer: $INSTALLER"
echo "Size: ${SIZE_MB} MB  (ceiling: ${CEILING} MB)  [DSK-03]"

if [ "$SIZE_MB" -gt "$CEILING" ]; then
    echo "::error::Installer ${SIZE_MB} MB exceeds ${CEILING} MB budget (DSK-03)"
    exit 1
fi

echo "OK: installer size within budget"
```

NOTE: `du -sm` returns megabytes (1 MB = 1024×1024 bytes on macOS, 1000×1000 on GNU du).
The difference is at most 5% and well within the 30 MB ceiling headroom for a ~10–15 MB installer.
Do not use `stat -c` (Linux-only) or `wc -c` (byte-level, harder to read). du -sm is portable
across macOS and Linux CI runners.

Step added for macOS installer size check:
```yaml
- name: "Footprint gate — DMG size < 30 MB (macOS)"
  if: runner.os == 'macOS'
  shell: bash
  run: |
    DMG=$(ls target/universal-apple-darwin/release/bundle/dmg/*.dmg 2>/dev/null | head -1)
    if [ -z "$DMG" ]; then
      echo "::error::No .dmg found — tauri-action may have failed"

```

Step added for Windows installer size check:
```yaml
- name: "Footprint gate — Installer size < 30 MB (Windows)"
  if: runner.os == 'Windows'
  shell: pwsh
  run: |
    $installer = Get-ChildItem -Path "target\x86_64-pc-windows-msvc\release\bundle" `
      -Recurse -Include "*.msi","*.exe" | Select-Object -First 1
    if (-not $installer) {
      Write-Error "No installer found — tauri-action may have failed"
      exit 1
    }
    $sizeMb = [math]::Ceiling($installer.Length / 1MB)
    Write-Host "Installer: $($installer.FullName)"
    Write-Host "Size: $sizeMb MB  (ceiling: 30 MB)  [DSK-03]"
    if ($sizeMb -gt 30) {
      Write-Error "Installer $sizeMb MB exceeds 30 MB budget (DSK-03)"
      exit 1
    }
    Write-Host "OK: installer size within budget"
```

Step added for RSS gate (Linux-only in the matrix — macOS runner also runs bench-rss; skip on Windows):
```yaml
- name: "Footprint gate — Idle RSS < 150 MB"
  if: runner.os != 'Windows'
  shell: bash
  run: |
    # Build the bench-rss binary (release profile; same build as foliom binary above).
    # tauri-action already ran cargo build --release internally, but foliom-bench-rss
    # is a separate bin target that tauri-action doesn't build. Build it explicitly.
    cargo build --release --bin foliom-bench-rss --locked

    # Generate a minimal 100-note corpus if the 5k corpus isn't available.
    # (The 5k corpus generation is only in the bench job of ci.yml, not here.)
    # For the RSS gate we just need a valid vault — use the synthetic fixture.
    CORPUS_PATH="crates/core/tests/fixtures/logseq-synthetic"

    # Override the RSS ceiling to 150 MB (DSK-03) and point at the release binary.
    FOLIOM_BENCH_CEILING_MB=150 \
    FOLIOM_BENCH_FOLIOM="target/release/foliom" \
      ./target/release/foliom-bench-rss "$CORPUS_PATH"
```

NOTE on RSS gate scope (from 05-RESEARCH Open Question 3): `foliom-bench-rss` measures the
`foliom serve` process RSS — the axum server + indexer + watcher. The Tauri WebView renderer
runs as a separate OS process (WebView2.exe / com.apple.WebKit.WebContent) and is NOT included
in this measurement. This is intentional for v1: the 150 MB budget covers the in-process backend;
WebView renderer memory is browser-vendor-controlled and excluded. Document in PERF-BASELINE.md.

The bench-rss binary uses `FOLIOM_BENCH_FOLIOM` env var to select the binary path (per plan 02-08
implementation). In the release workflow, the foliom binary is at `target/release/foliom`
(not the Tauri bundle). This is correct — bench-rss measures the headless serve process RSS,
which is identical whether launched via `foliom serve` or from inside Tauri.
