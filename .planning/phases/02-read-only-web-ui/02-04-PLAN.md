<interfaces>
Rust-side reference for `stripForRender` semantics — TypeScript implementation must mirror this behavior so that backend and frontend agree on what gets rendered:

`crates/core/src/parser/ast.rs::strip_segmenter_prefix` strips:
- Leading TABs (count = depth)
- First-line `- ` bullet marker
- Continuation-line `  ` (2 spaces) marker
- Property lines (`key:: value` where key matches `[A-Za-z][A-Za-z0-9._-]*`)
- Drawer ranges (`:NAME:` through `:END:` inclusive)

Plus, in this plan, an inline reference to the markdown-it heading-rule problem (Pitfall 2 / Open Question 1): markdown-it runs inline rules inside ATX headings by default, but we must NOT extract chips from heading content (PRS-04 invariant). Plan 02-04 Task 1 includes a 30-min spike to choose between three mechanisms and lock the choice.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `raw` field from backend → `md.render` → `{@html ...}` | `markdown-it({ html: false })` strips inline HTML; rule output uses our own `escapeHtml`; Prism output is HTML built from `Prism.highlight` (token names from a fixed grammar, content escaped) |
| `data-page` / `data-tag` attribute → `push(/pages/...)` | Attribute values are already `escapeHtml`-encoded; `encodeURIComponent` in the click handler re-encodes on the way out |
| Deep-link URL fragment → DOM lookup | `block=N` parsed as integer; `document.getElementById('block-' + N)` is safe (only a digit string) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-12 | XSS | `{@html md.render(...)}` | mitigate | `markdown-it({ html: false })` explicitly set; inline-rule render hooks escape every interpolated user value via `escapeHtml`; covered by `rules.test.ts` asserting `<` and `>` escape correctly in chip content. |
| T-02-13 | XSS | Prism highlight output | mitigate | Prism's grammar-based highlighter produces a fixed set of token classes; user code content is escaped by Prism itself. Verified by passing a `<script>` payload through a code fence in vitest and asserting it does not produce a `<script>` element. |
| T-02-14 | DoS | 1000+ block render | mitigate | Soft perf budget tested in `block.test.ts`. If exceeded, virtualize (deferred to v1.x per A7). |
| T-02-15 | DoS | Deeply nested block tree | mitigate | Synthetic depth-limited fixtures in tests; backend already caps via Phase 1 parser depth handling. No additional gate in this plan. |
| T-02-16 | Tampering | URL fragment `#block=N` | accept | Worst case is scrolling to the wrong element or a no-op. No security implication for a single-user local app. |
</threat_model>
